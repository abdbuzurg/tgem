# Phase 1 — Schema diff and migration plan

## TL;DR

The legacy production schema and the rewrite's frozen `00001_initial_schema.sql` baseline are **structurally identical** (50 tables match, modulo SQL keyword-quoting on one column). The rewrite's real schema evolution lives in migrations `00002`–`00006`, which is what we will actually apply against production.

**Migration plan, one paragraph:** stamp `goose_db_version` to mark `00001` as already applied (because it *is* — it is the legacy AutoMigrate output frozen as SQL), then let the rewrite container's embedded `MigrateUp` apply `00002`-`00006` at first startup. No dropped columns. No lossy type changes. No rename-with-data-preservation choreography. One row-level cleanup (`00002`) that is lossy by design but only touches orphan rows or already-broken data.

The detailed analysis follows.

---

## 1. Methodology and caveats

**What was diffed:** GORM model structs at `_legacy/tgem-backend/model/*.go` (the AutoMigrate list in `database.go:48-99` is authoritative) against the full SQL of `tgem-backend-rewrite/internal/database/migrations/00001_initial_schema.sql` (3000 lines, generated by running AutoMigrate against the *rewrite*'s own models and dumping with `pg_dump --schema-only`).

**What was NOT diffed:** the actual schema in the production database. We can't read it from here. The legacy uses GORM's `AutoMigrate` at every startup, which is **additive only** — it adds tables and columns but never drops them. So if any legacy model was deleted or any field renamed/removed *during the lifetime of the deployed code*, the live database still has the old shape lurking.

**Implication:** this diff is correct *for the bytes of code in `_legacy/`*. It is a strong proxy for what is in production, but not a proof. The runbook (Phase 3) starts with `pg_dump --schema-only` against production and a textual diff against `00001_initial_schema.sql`. If extra tables/columns show up, they are harmless residue and can be left in place (rewrite code ignores them); if expected tables/columns are missing, that is a **STOP** condition and Phase 1 needs to be revisited.

**Type-mapping conventions** used by both sides via GORM:

| Go type            | Postgres type                  |
| ------------------ | ------------------------------ |
| `uint`             | `bigint NOT NULL`              |
| `*uint`            | `bigint` (nullable)            |
| `string`           | `text`                         |
| `bool`             | `boolean`                      |
| `time.Time`        | `timestamp with time zone`     |
| `decimal.Decimal`  | `numeric(precision, scale)`    |
| `float64`          | `numeric`                      |

`tinyText` GORM tag is informational only on PostgreSQL — it stays `text`. Primary keys are always `bigint` with an attached `id_seq` sequence.

### 2.3 Removed in rewrite at baseline

**`material_providers`** — exists as a model file at `_legacy/tgem-backend/model/material_provider.go` but **is not in the AutoMigrate list in `database.go:48-99`.** It is dead code in the legacy. If it exists in the production database, it is either AutoMigrated by an older deployment and now ignored, or seeded manually. The rewrite has no reference to it and will not query it.

Treatment in this migration: **leave the table in place untouched.** It is harmless residue. Rollback: not applicable (we don't drop it).

### 2.4 Modified in rewrite at baseline

**`worker_attendances`** — column `end` is *quoted* in the baseline SQL (`"end" timestamp with time zone`) because `end` is a Postgres reserved keyword. The legacy Go struct field `End` AutoMigrates to the same column without quotes (`end timestamp with time zone`). Functionally identical — the column is the same column. GORM and the pgx driver both quote reserved identifiers when writing queries, so no SQL on either side is sensitive to this difference.

Treatment: **no change.** Not actually a diff.

None.

None from the code diff. The only *real* uncertainty is production drift (§1 caveat).

## 3. What each later migration does to a production database

These will all run automatically when the rewrite container starts for the first time post-stamp.

### `00002_phase7_data_cleanup.sql` — phase-7 data cleanup

**Schema changes:**
- Adds `CREATE UNIQUE INDEX IF NOT EXISTS auction_participant_prices_item_user_uq ON auction_participant_prices (auction_item_id, user_id)` — required by post-migration `ON CONFLICT` paths.

**Data changes (lossy by design):**
- Deletes orphan rows in `objects` whose `object_detailed_id` no longer points at a row in the corresponding detail table (`kl04_kv_objects`, `mjd_objects`, `s_ip_objects`, `stvt_objects`, `tp_objects`, `substation_objects`, `substation_cell_objects`). These orphans were created by historical Delete bugs (per the migration's own commentary). On a clean database with no bug history, the deletes are no-ops.
- Re-tags misclassified `tp_nourashes_objects.target_type` rows from `'kl04kv_objects'` to `'mjd_objects'` where the `target_id` actually points at an mjd object's parent.
- Deletes orphan `tp_nourashes_objects` rows whose `sip_objects` target was deleted by the buggy Delete.
- Deletes orphan `object_supervisors` and `object_teams` rows whose `object_id` no longer exists.
- Coalesces duplicate `material_locations` rows for `location_type='out-of-project'`: sums amount across duplicates, keeps the lowest id, deletes the rest. Required by a former `invoice_output_out_of_project` Confirm bug.
- Backfills missing `invoice_counts` rows for every (project, invoice_type) pair, with `count` equal to the existing invoice count of that type. The shared `'output'` counter sums both `invoice_outputs` and `invoice_output_out_of_projects`.

**Reversibility:** the migration's own Down block only drops the unique index. The data cleanup is irreversible by SQL alone — orphan rows are gone, duplicate rows are coalesced. The only way to restore the pre-migration state is `pg_restore` from the pre-migration backup. This is acceptable because (a) the deleted rows were unreachable from the application code anyway, and (b) the backup taken in runbook step 3.2 is the explicit rollback path.

### `00003_align_correction_resource_url.sql` — one-row data fix

Updates a single seeded `resources` row: `WHERE name='Корректировка оператора' AND url='/invoice-correction'` → set `url='/correction'`. Idempotent guard on the WHERE clause. Down is the inverse. Trivially reversible.

### `00004_split_output_out_of_project_counter.sql` — counter seed

Inserts an `invoice_counts` row with `invoice_type='output-out-of-project'` and `count=0` for every existing project that doesn't already have one. Idempotent. Down deletes all `invoice_counts WHERE invoice_type='output-out-of-project'` — safe if no out-of-project invoices have been issued post-migration; lossy otherwise.

### `00005_permissions_v2_foundation.sql` — permissions-v2 schema

**Schema additions** (none of these tables existed before; legacy never had them):
- `resource_types(code text PK, category, display_ru NOT NULL, display_en)`
- `permission_actions(code text PK, display_ru NOT NULL)`
- `roles.code` column (nullable initially, then backfilled and set NOT NULL UNIQUE)
- `role_grants(role_id, resource_type_code, action_code, PK on all three, all FKs)`
- `user_roles(id serial PK, user_id, role_id, project_id nullable, granted_by, granted_at)` with partial unique indexes split by `project_id IS NULL` vs not.

**Data additions:**
- Seeds 9 `permission_actions` rows.
- Seeds 40+ `resource_types` rows including the new `admin.user_action` for the audit log.
- Backfills `roles.code` for the 8 canonical role names (`superadmin`, `warehouse_keeper`, `pto`, `bidder`, `supply_officer`, `supervisor`, `regional_project_manager`, `project_manager_assistant`). Roles whose name doesn't match the canonical patterns get a stable fallback `role_<id>`.
- Locks `roles.code` to NOT NULL UNIQUE.
- Seeds `role_grants` for the 8 canonical roles per the permissions spec.
- Backfills `user_roles` from `users.role_id` as one global (`project_id IS NULL`) row per user.

**Compatibility:**
- The legacy `permissions` and `resources` tables stay in place untouched. Per the migration's own commentary, the v2 tables run *alongside* the legacy ones; the legacy tables remain authoritative through phase 4 of the rewrite roadmap (we are not yet at phase 4). This means logged-in users' effective permissions don't change at cutover — the frontend keeps reading the same legacy permission shapes via the same API surface.
- The new `roles.code` column is read by post-cutover code, but at cutover-instant nothing references it. Safe.

**Down block:** drops the 4 new tables, drops `roles.code`. Safe and complete reversal *of the schema*. Does not undo the implicit assumption "v2 is being rolled out" anywhere — there is nothing else to undo.

**Risk:** the `UPDATE roles SET code = ... WHERE name = ...` block matches on Russian-language `roles.name` values. If production has hand-edited role names that don't match the canonical patterns, those roles get the `role_<id>` fallback. That's a cosmetic concern (the admin UI in phase 3 will let operators rename), not a functional one — the v1 permission system still works.

### `00006_user_action_audit.sql` — audit-log expansion

Adds `http_method text` and `request_ip text` columns to the existing `user_actions` table (both nullable, both `ADD COLUMN IF NOT EXISTS`). Creates three indexes (`date_of_action DESC`, `(user_id, date_of_action DESC)`, `(project_id, date_of_action DESC)`).

Cleanly reversible. Backfill: not done — old `user_actions` rows just have NULL for the new columns, which the audit-list UI handles.

## 4. The actual migration plan

### 4.1 Pre-flight verification (must pass before anything is applied)

Run against production:

```sql
-- Tables that must exist (50 from 00001 baseline). One row per table; should return 50.
SELECT count(*) FROM information_schema.tables
WHERE table_schema = 'public'
  AND table_name IN (
    'auction_items','auction_packages','auction_participant_prices','auctions',
    'districts','invoice_counts','invoice_inputs','invoice_materials',
    'invoice_object_operators','invoice_objects','invoice_operations',
    'invoice_output_out_of_projects','invoice_outputs','invoice_returns',
    'invoice_write_offs','kl04_kv_objects','material_costs','material_defects',
    'material_locations','materials','mjd_objects','object_supervisors',
    'object_teams','objects','operation_materials','operations',
    'operator_error_founds','permissions','project_progress_materials',
    'project_progress_operations','projects','resources','roles','s_ip_objects',
    'serial_number_locations','serial_number_movements','serial_numbers',
    'stvt_objects','substation_cell_nourashes_substation_objects',
    'substation_cell_objects','substation_objects','team_leaders','teams',
    'tp_nourashes_objects','tp_objects','user_actions','user_in_projects',
    'users','worker_attendances','workers'
  );

-- Goose tracking table must NOT exist yet. Should return 0.
SELECT count(*) FROM information_schema.tables
WHERE table_schema = 'public' AND table_name = 'goose_db_version';

-- Permissions-v2 tables must NOT exist yet. Should return 0.
SELECT count(*) FROM information_schema.tables
WHERE table_schema = 'public'
  AND table_name IN ('resource_types','permission_actions','role_grants','user_roles');

-- roles.code column must NOT exist yet. Should return 0.
SELECT count(*) FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = 'roles' AND column_name = 'code';

-- user_actions.http_method must NOT exist yet. Should return 0.
SELECT count(*) FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = 'user_actions' AND column_name = 'http_method';
```

If any of these fail, **STOP** and investigate before stamping or migrating.

A more thorough check (recommended on cutover day): `pg_dump --schema-only` against production into `live_schema.sql`, run the rewrite's `00001_initial_schema.sql` against a fresh empty database into `expected_schema.sql` (or have a previously-captured one ready), then `diff` them. Acceptable diffs are: column ordering, ownership, and sequence default expressions. Unacceptable: any structural difference in table or column existence.

### 4.2 Apply the migration

The full mechanic, in order. SQL goes in `01-migrations.sql`. The actual commands the operator runs go in the Phase 3 runbook.

1. **Stamp Goose** — create the `goose_db_version` tracking table and mark versions 0 and 1 as applied. This tells the rewrite's embedded `MigrateUp` "00001 is done, don't try to run it again." The rewrite container will only consider 00002..00006 on startup.

2. **Start the rewrite backend container.** Its `MigrateUp` runs automatically and applies 00002..00006 in order. The `IF NOT EXISTS` guards and the per-migration idempotency comments mean a partial-then-retry is safe.

3. **Verify** — sanity queries against the post-migration state (row counts in the new tables, presence of the indexes, the corrected resource URL).

Alternative: apply migrations *out-of-band* using the `goose` CLI on the host before starting the rewrite container. This is more cautious (migration outcome known before any service starts) but requires installing the goose binary. The runbook in Phase 3 goes with the simpler "stamp + let container migrate" path; an explicit goose-CLI variant is documented as a fallback there.

### 4.3 Post-migration verification queries

```sql
-- Goose should now show version 6.
SELECT MAX(version_id) FROM goose_db_version WHERE is_applied;
-- expected: 6

-- The new tables exist.
SELECT count(*) FROM information_schema.tables
WHERE table_schema = 'public'
  AND table_name IN ('resource_types','permission_actions','role_grants','user_roles');
-- expected: 4

-- Permissions actions and resource types are seeded.
SELECT count(*) FROM permission_actions;       -- expected: 9
SELECT count(*) FROM resource_types;            -- expected: 40

-- Every user with a role got a global user_roles row.
SELECT count(*) FROM user_roles WHERE project_id IS NULL;
-- expected: same as SELECT count(*) FROM users WHERE role_id IS NOT NULL;

-- The output-out-of-project counter row exists for every project.
SELECT count(*) FROM invoice_counts WHERE invoice_type = 'output-out-of-project';
-- expected: same as SELECT count(*) FROM projects;

-- The correction resource URL is the new value.
SELECT count(*) FROM resources WHERE name = 'Корректировка оператора' AND url = '/correction';
-- expected: 1 (or 0 if production had a different name; that's a separate concern)

-- user_actions has the new columns.
SELECT column_name FROM information_schema.columns
WHERE table_schema = 'public' AND table_name = 'user_actions'
  AND column_name IN ('http_method','request_ip');
-- expected: 2 rows
```

## 5. BLOCKERS

These are the conditions under which the runbook must **STOP** rather than proceed.

1. **Pre-flight table count is not 50.** Either tables exist that we didn't model (extra residue from legacy AutoMigrate — actually fine, investigate but don't stop) or tables are missing (real problem, stop).

2. **Pre-flight finds `goose_db_version` already exists.** Production has already been migrated, or someone else stamped it. Stop, read its contents, decide.

3. **Pre-flight finds permissions-v2 tables already exist (`resource_types`, `permission_actions`, `role_grants`, `user_roles`).** Migration has already been partially applied. Stop, run the verification queries (§4.3), decide whether to skip the stamp and let MigrateUp resume.

4. **A `pg_dump --schema-only` diff against the rewrite's baseline produces column-level differences.** Examples that *would* be blockers: a column expected by `00005` (`users.role_id`, `roles.id`, `roles.name`) has a different type than the rewrite expects. Examples that *would not* be blockers: an extra column on `users` from a since-removed legacy model field — the rewrite ignores it.

5. **Cutover during the 23:55 Asia/Dushanbe cron window.** The legacy backend runs the same job (`progress_report_daily`). If we cut over while it's mid-run, we risk a half-completed snapshot. The runbook says: do the cutover before 23:00 or after 00:30 local Dushanbe time.

No data-shape blockers were found in the code-only diff (no NOT NULL column without a sensible default, no narrowing type changes, no FK target deletions).
